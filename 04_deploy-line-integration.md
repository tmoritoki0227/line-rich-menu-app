## ステップ13：S3 バケットの作成（フロントエンドホスティング用）

Vue 3 アプリの静的ファイル（HTML / CSS / JS）を保存する S3 バケットを作成する。

### 13-1. バケットの作成

1. AWSコンソール ＞ **S3** ＞ **[バケットを作成]** をクリック。
2. 以下を入力する。

    | 項目 | 値 |
    |---|---|
    | バケット名 | `line-rich-menu-app` |
    | リージョン | 東京（ap-northeast-1） |

    > **バケット名はグローバルで一意**

1. **「パブリックアクセスをすべてブロック」** は **ON のまま**にする。

    > **なぜパブリックアクセスをブロックするか**: S3 を直接公開せず、CloudFront 経由でのみアクセスさせることでセキュリティを高める。CloudFront の OAC（後述）が S3 への唯一のアクセス経路になる。

4. 他の設定はデフォルトのまま ＞ **[バケットを作成]**。

## ステップ14：CloudFront ディストリビューションの作成

CloudFront は AWS の CDN（コンテンツ配信ネットワーク）サービス。S3 の静的ファイルを世界中のエッジロケーションからユーザーに高速配信する。LIFF のエンドポイント URL に使用する。

### 14-1. ディストリビューションの作成

1. AWSコンソール ＞ **CloudFront** ＞ **[ディストリビューションを作成]** をクリック。

2. プランの選択画面が表示されたら **[Pay as you go]** を選択して次へ進む。

   > **Pay as you goとは**: 使った分だけ課金される従量課金プラン。個人用の軽い利用では月に数十円〜数百円程度。

3. **[Distribution name]** に `kakeibo-distribution` と入力する（任意の名前でよい）。

4. **[Distribution type]** で **[Single website or app]** を選択し、**[Next]** をクリック。

5. **[Origin type]** で **[Amazon S3]** を選択する。

6. **[S3 origin]** の入力欄をクリックし、ステップ13で作成したバケット名を選択する。
   - 例: `my-kakeibo-liff-bucket-2026.s3.ap-northeast-1.amazonaws.com`

7. **[Allow private S3 bucket access to CloudFront]** を **[Yes]** にチェックする。

   > **これが OAC の設定**: CloudFront だけが S3 にアクセスできるようにする署名付き認証の仕組み（Origin Access Control）。S3 バケットを非公開のまま CloudFront 経由でのみ配信できる。

8. **[Web Application Firewall (WAF)]** で **[Do not enable security protections]** を選択する。

   > WAF は追加料金が発生する。個人開発での検証には不要。

9. **「一般」** の設定で **[Default root object]** に `index.html` と入力する。

   > **デフォルトルートオブジェクトとは**: `https://xxxx.cloudfront.net/` のようにパスを指定しないでアクセスしたときに返すファイル。`index.html` にすることで Vue アプリのトップページが開く。

10. **[ディストリビューションを作成]**（または **[Next]** ＞ **[Create]**）をクリック。

11. 作成完了後、**オリジンの編集画面**（またはディストリビューション詳細 ＞ **[オリジン]** タブ ＞ オリジンを選択 ＞ **[編集]**）を開き、**[ポリシーをコピー]** ボタンをクリックする。

12. **[変更を保存]** をクリックする。

### 14-2. S3 バケットポリシーの更新

1. S3 コンソール ＞ 作成したバケット ＞ **[アクセス許可]** タブ。
2. **「バケットポリシー」** の **[編集]** をクリック。
3. ステップ14-1でコピーしたポリシーを貼り付けて **[変更の保存]** をクリック。

### 14-3. カスタムエラーレスポンスの設定

> **SPA（シングルページアプリ）とは**: Vue 3 で作ったアプリはすべてのページ遷移をブラウザ側の JavaScript で制御する。サーバー（S3）には `index.html` しかなく、`/about` などのパスに直接アクセスすると S3 が「ファイルなし」として 403/404 を返してしまう。この設定を入れることで「どんなパスへのアクセスでも `index.html` を返す」ようにし、Vue がクライアント側でルーティングを処理できるようにする。

1. 作成したディストリビューション ＞ **[エラーページ]** タブ ＞ **[カスタムエラーレスポンスを作成]** をクリック。
2. 以下の内容を入力する。

   | 項目 | 値 |
   |---|---|
   | HTTP エラーコード | `403: Forbidden` |
   | エラーレスポンスをカスタマイズしますか? | **はい（Customize error response）** |
   | レスポンスページのパス | `/index.html` |
   | HTTP レスポンスコード | `200: OK` |

3. **[変更を保存]** をクリック。
4. **同じ手順をもう一度** 繰り返し、HTTP エラーコード `404: Not Found` にも同じ設定を追加する。

### 14-4. ドメイン名の確認

ディストリビューション一覧で **「ドメイン名」** 列に表示される URL（例: `https://xxxxxxxxxxxx.cloudfront.net`）をコピーして保存する。ステップ15・16 で使用する。

> デプロイのステータスが「デプロイ中」から「有効」になるまで数分かかることがある。

## ステップ15：フロントエンドのビルドと初回デプロイ

### 15-1. API URL の最終確認

`frontend/src/constants.ts` の `API_BASE_URL` が記事③で取得した API Gateway の URL になっていることを確認する。

```typescript
export const API_BASE_URL = 'https://xxxxxxxxxx.execute-api.ap-northeast-1.amazonaws.com/dev'
```

### 15-2. フロントエンドのビルド

```bash
# line-rich-menu-app/frontend フォルダで実行
cd frontend
npm run build
```

成功すると `frontend/dist/` フォルダに静的ファイルが生成される。

```
frontend/dist/
├── index.html
├── assets/
│   ├── index-xxxxxx.js   ← Vue アプリの JavaScript（圧縮・バンドル済み）
│   └── index-xxxxxx.css  ← Tailwind CSS（圧縮済み）
└── ...
```

> **静的ファイルとは**: ビルドで生成された HTML / CSS / JavaScript ファイルのこと。サーバーで処理を行わず、そのままブラウザに送ることで動く。S3 に置いて CloudFront で配信することで、サーバーレスなフロントエンドホスティングが実現できる。

### 15-3. S3 へのアップロード

> **⚠️ SSO セッションの確認**: AWS CLI のSSO セッションは約 8 時間で期限切れになる。アップロード前に必ず認証状態を確認すること。

```bash
# SSO セッションが切れていたらログインし直す
aws sso login --profile line-rich-menu-app

# 認証確認（自分のアカウント ID が表示されれば OK）
aws sts get-caller-identity --profile line-rich-menu-app
```

`dist/` フォルダを S3 にアップロードする（`frontend/` 直下で実行する）。

```bash
aws s3 sync dist/ s3://YOUR_BUCKET_NAME --delete --profile line-rich-menu-app
```

> **`dist/` を指定する理由**: `npm run build` の出力は `frontend/dist/` に入っている。S3 に上げるのはこの中の HTML / CSS / JS ファイルで、`frontend/` フォルダごとではなく `dist/` の**中身だけ**を同期する。
>
> **`--delete` オプション**: S3 側に古いファイルが残らないよう、ローカルに存在しないファイルを S3 から削除する。

### 15-4. CloudFront キャッシュの削除と確認

S3 を更新しただけでは CloudFront のキャッシュが残っているため、最新版が配信されない場合がある。以下でキャッシュを削除する。

```bash
aws cloudfront create-invalidation \
  --distribution-id YOUR_DISTRIBUTION_ID \
  --paths "/*" \
  --profile line-rich-menu-app
```

キャッシュ削除の完了には数十秒かかる。少し待ってからブラウザで確認する。

ブラウザで `https://xxxxxxxxxxxx.cloudfront.net` を開くと、「読み込み中...」の後に「LIFF エラー」画面が表示される。これは**正常**。

> **なぜ LIFF エラーが出るか**: LIFF アプリに登録されているエンドポイント URL がまだ `https://example.com`（ダミー）のまま。次のステップ 16-1 で CloudFront の URL に更新する。

## ステップ16：LINE 設定の更新

### 16-1. LIFF エンドポイント URL の更新

記事②で LIFF アプリに登録したエンドポイント URL（ダミーの `https://example.com`）を、CloudFront の URL に更新する。

1. [LINE Developers コンソール](https://developers.line.biz/console/) ＞ `勉強会用-v3` ＞ `LIFF-v3` チャネル ＞ **[LIFF]** タブ。
2. `入力フォーム-v3` の **[編集]** をクリック。
3. **「エンドポイントURL」** を CloudFront の URL に変更する。

```
https://xxxxxxxxxxxx.cloudfront.net
```

4. **[更新]** をクリック。

### 16-2. LINE Webhook URL の登録

LINE からのメッセージを Lambda で受け取るために、Webhook URL を登録する。

1. LINE Developers ＞ `勉強会用-v3` ＞ `初めてのLINEボット-v3` ＞ **[Messaging API設定]** タブ。
2. **「Webhook URL」** の **[編集]** をクリック。
3. API Gateway の `/webhook` エンドポイントを入力する。

```
https://xxxxxxxxxx.execute-api.ap-northeast-1.amazonaws.com/dev/webhook
```

4. **[更新]** ＞ **[検証]** をクリック。
   - 「成功」と表示されれば正常に Lambda に接続できている。
5. **「Webhookの利用」** を **ON** に切り替える。

## ステップ17：LINE Official Account の設定

### 17-1. プロフィールアイコンの変更（任意）

デフォルトのアイコンをカスタム画像に変えることで、トーク画面での見栄えが良くなる。

1. [LINE Official Account Manager](https://manager.line.biz/) にログイン ＞ `初めてのLINEボット-v3` を選択。
2. 左メニュー **[ホーム]** ＞ 右上の **[設定]** をクリック。
3. 左メニュー **[アカウント設定]** をクリック。
4. **「基本情報」** セクション ＞ アイコン画像の **[変更]** をクリック。
5. 使いたい画像をアップロード（JPG / PNG 推奨、正方形が見栄え良い）＞ **[適用]** ＞ **[保存]**。

### 17-2. リッチメニューの作成

リッチメニューとは LINE のトーク画面下部に常時表示できるメニュー。2行×3列の6分割レイアウトで、以下の6つのアクションを割り当てる。

#### ボタン画像の準備（Canva が便利）

リッチメニューには画像を貼ることができ、各ボタンにラベルやアイコンを入れると視認性が上がる。**[Canva](https://www.canva.com/)** を使うと無料で簡単に作成できるのでおすすめ。

**Canva での作成手順:**
1. Canva を開き、**[カスタムサイズ]** で `2500 × 1686 px` のデザインを作成する。
2. 2行×3列のガイド線を引いて6つのエリアに分割する。
3. 各エリアに以下のラベルを入れる。

| エリア | 推奨ラベル（画像に書く文字） | プログラムとの対応 |
|---|---|---|
| A（左上） | 📝 入力フォーム | LIFF フォームを開く |
| B（中上） | 📊 今月の合計 | 「合計」を送信 → 当月合計を返す |
| C（右上） | 📋 最新の履歴 | 「最新の履歴を表示」を送信 → 最新5件を返す |
| D（左下） | 🐱 今日の猫 | 猫画像サイトへのリンク |
| E（中下） | ❓ ヘルプ | 「ヘルプ」を送信 → 操作メニューカードを返す |
| F（右下） | 🚧 機能2 | 「機能2」を送信 → 準備中メッセージを返す |

> **ポイント**: タイプが「テキスト」のボタンは、設定した文字列がそのまま LINE に送信されてプログラムが受け取る。画像のラベルはあくまで見た目用なので、プログラムが動くかどうかはテキスト設定の値が正しいかで決まる。

4. デザインが完成したら **[ PNG でダウンロード]** する。

**作成手順:**

1. [LINE Official Account Manager](https://manager.line.biz/) ＞ `初めてのLINEボット-v3` ＞ 左メニュー **[ホーム]** ＞ **[リッチメニュー]** ＞ **[作成]** をクリック。

2. **「表示設定」** セクション:
   - タイトル: `メニュー`
   - 表示期間: 開始日を「今日」に設定 ＞ 終了日は設定しない（制限なし）

3. **「コンテンツ設定」** ＞ **[テンプレートを選択]** をクリック ＞ **6分割（2行×3列）** のレイアウトを選択 ＞ **[適用]**。

4. **リッチメニュー画像のアップロード**:
   - **[画像をアップロード]** をクリックし、Canva でダウンロードした PNG をアップロードする。
   - 画像を使わない場合はテキスト設定だけでもよい。
   - 推奨サイズ: 2500 × 1686 px（LINE 公式の推奨サイズ）。

5. **エリア A（左上）** をクリックして設定する。
   - タイプ: `リンク`
   - URL: `https://liff.line.me/（LIFF ID）`
   - アクションラベル: `入力フォーム`

   > **LIFF URL の確認方法**: LINE Developers ＞ `LIFF-v3` ＞ LIFF タブ ＞ 発行された `https://liff.line.me/〜` の URL をコピー。

6. **エリア B（中上）** をクリックして設定する。
   - タイプ: `テキスト`
   - テキスト: `合計`

   > **テキストタイプについて**: タイプを「テキスト」にするとアクションラベルの入力欄はなく、テキストフィールドに入れた文字がそのまま送信される（かつボタンのラベルにもなる）。

7. **エリア C（右上）** をクリックして設定する。
   - タイプ: `テキスト`
   - テキスト: `最新の履歴を表示`

8. **エリア D（左下）** をクリックして設定する。
   - タイプ: `リンク`
   - URL: `https://cataas.com/cat`
   - アクションラベル: `今日の猫`

9. **エリア E（中下）** をクリックして設定する。
   - タイプ: `テキスト`
   - テキスト: `ヘルプ`

   > **動作**: 「ヘルプ」と送信するとボット側で Flex Message（操作メニューカード）を返す。カード内のボタンをタップするとキーワードがチャットに送信される。

10. **エリア F（右下）** をクリックして設定する。
    - タイプ: `テキスト`
    - テキスト: `機能2`

    > **動作**: 「機能2」と送信すると「🚧 この機能は現在準備中です。」と返る。未実装機能のプレースホルダーとして使う。

11. **[保存]** をクリックし、確認画面で **[公開]** をクリック。

> **動作確認**: スマホで LINE を開き、`初めてのLINEボット-v3` アカウントのトーク画面下部に6分割のリッチメニューが表示されていれば成功。

## ステップ19：友だちへの公開

> **自分だけで使う場合はこのステップ不要。** 自分のスマホで友だち追加して使うだけなら、開発中ステータスのままで問題なく動く。家族や友人にも使ってもらいたい場合のみ実施する。

自分以外の人がボットを使えるようにするには、LIFF チャンネルの公開ステータスを変更する必要がある。デフォルトでは「開発中」状態のため、管理者（自分）以外は LIFF アプリ（入力フォーム）にログインできない。

> **Messaging API チャンネルにはステータスがない。** ボットへの友だち追加は開発中のままでも誰でも可能。公開が必要なのは **LINEログインチャンネル（LIFF用）** のみ。

### 19-1. LIFF チャンネルを公開状態にする

1. [LINE Developers](https://developers.line.biz/) ＞ プロバイダー（例: `勉強会用-v3`）を選択する。
2. プロバイダー内のチャンネル一覧から **`LIFF-v3`**（**LINEログイン**と表示されているもの）をクリックして開く。
3. チャンネル設定画面が開いたら **[チャンネル基本設定]** タブをクリックする。
4. ページ下部の **「チャンネルステータス」** を確認する。
   - `開発中` と表示されている場合は隣の **[公開する]** ボタンをクリックする。
   - 確認ダイアログが出たら **[公開]** をクリック。
5. ステータスが **`公開済み`** に変わったことを確認する。

> ⚠️ **この操作は元に戻せない。** 公開済みから開発中に戻すにはチャンネルを削除して再作成するしかない。本当に共有する準備ができてから実施すること。

> **「開発中」と「公開済み」の違い**: 開発中は管理者と招待されたテスターのみが LIFF アプリにログインできる。公開済みにすることで LINE ユーザー全員が LIFF アプリ（入力フォーム）を使えるようになる。
